Preamble

With software development favoring open to the public, we have a variety of audiences to cater for. Now many of
us in computer science can understand the concepts and how software operates after the initial build. However,
we must also be inclusive about others who may only have generic technical expertise.

Our documentation will serve two purposes: we want our work to be easy to understand for users of all levels in
technology, but also encourage others to step into the field.

Introduction

Any project you create should always have an introduction. The introduction doesn't have
to be anything long or verbose. As long as you can succiently convey the information for the
user to understand, that should solve it.

Table of Contents

Having a table of contents provides multiple benefits when completed. A user can see just
what everything is included in the readme. The user can click on the sections (like the toc I provided above)
and can then click back to the previous screen. I included this mainly due to not
wanting to scroll through everytime I needed the right information. Usually the skeleton will look like this:

[Go here](#gohere)

And this will include a < div> tag to link the section.

Summary

As stated above, include a summary of what your project can do. A summary
may also have to be added to sections in case users have questions. The summary
can also be included in the introduction if you don't think a closing doesn't needed to be provided.

Sections

One argument many generic users experience is that not having sections
properly dedicated to certain steps or when switching off to a different
program. This is usally in part that we think a user may know what to do intuitively.

When we break up parts into sections, we can be safe to assume the user
won't get lost or have to backtrack. This helps create a README that a user
can use just like they are reading a book.

Prerequisites

The prerequisites should always be included before beginning the project
gets underway. Theres no point continuing if the right ingredients aren't included.
When adding this section, this should be the second section right after the introduction/summary.

In the prerequisites, also make sure you include what operating system is
being used for testing (Windows/Mac/Linux) before continuing down the road.

Steps

When including steps for following a procedure, keep only one step at a time.
Sometimes steps get conglomorated into one step that can throw a user off. For example:

  • Upgrade the software to 3.7.6, Accept all defaults, and then continue.

Though many of us can follow that, its best practice to keep the steps as one at a time.
This can also be called one-sentence-at-a-time. Plus depending on the hardware/sorftare,
the user may experience something else as "default"

Using Bullet Points

Bullet points I would only would like to use one source.

  • step one
  • step two
  • step three

When we start adding more bullet points underneath, it starts adding pressure
to keep moving around on the document instead of giving them a clear starting point.
We want the user to either move on to the next bullet point or to continue reading
the same bullet point followed by a period.

The bullet points aren't necessarily the main point but keeps the format in line.
The user's eyes know where to start on the page by the second bullet point. From there
a user will be able to pick up where they left off and can continue with their instructions.

Figures

Adding diagrams for users will help deomnstrate a point. This also correleates to if
the user gets lost, all they have to do is make sure the correct figure appears on screen.

When adding a figure

  • add italics underneath naming the figure with a one line description as example

Figure 1: Google's Fourth of July Banner
  • (Optional) Add an 'title' tag that can appear when a user hovers over the image.
    This might go into a longer explanation for the image or where to look at in question.

Figure 2: Google's Foruth of July Banner updated

Note: Adding to the visual representation, a figure placement can also
breakup the eye drain users experience looking thorough the documentation.

Notes

When adding notes, always keep it in bold but spell note like 'Note' not 'NOTE'.
I have found that users will go to that note first before anything else. Add the notes as such:

Note: Read the documentation before asking any questions

In this case, the note isn't pulling the reader away from the documentation but
reminds them this is important to read before continuing.

Code Blocks

When you include a use for code blocks, I would like code blocks to be used only
for code blocksand not to be used consecutively with folders or filepaths.

``` if this == true: print ("You are correct") ```

When you make a code block, wrap the entire piece with three (3) ```
symbols on top and bottom of the text.

FAQ

Frequently asked questions should be included at the end and be linked
into sections we assume users might have difficulty understanding. It doesn't
matter how good your software/project is, there will always be curious users.

This didn't work, what should I do?
  • Try something different

Issue History

As issues and bugfixes arrive with your software, you will want to add
what has changed with each different version that comes out. Formatting for
this section may look like this:

July 3, 2019

Discovered a new error in the testing software. It kept kicking out
the user instead of logging them in. Made the following corrections.

  • Corrected the said bug.
  • Developed a new interface.
